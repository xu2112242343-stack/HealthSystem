# 设计思路

## 这个项目在解决什么问题

慢性病的威胁往往藏在日常里，早期没有明显症状，等到发现时已经错过了最佳干预窗口。普通人缺少一个随手可用的工具来感知自己的风险水平，医生面对大量患者时也需要更高效的方式去做分层和随访，而平台运营方则需要一套机制来持续维护健康内容和医院资源。

HealthSystem 把这三件事串到了一起：**风险早筛 → 干预指导 → 就医推荐**。它不是要替代医生的判断，而是在医患之间搭一层缓冲，让用户先知道自己该关注什么，让医生看到更有结构的患者画像，让管理者能独立维护内容和数据。

项目面向三类使用者，各自场景不同：

- **普通用户**：填一份健康问卷，看到自己的三病风险（糖尿病、肝病、脑卒中）和综合健康分，拿到对应的饮食运动建议，必要时找到附近的医院。
- **医生**：查看患者群体的风险分布，钻取单个患者的详细档案，做随访记录和趋势对比。
- **管理员**：维护医生和用户账户，管理健康文章库和医院信息库，让内容不再依赖开发人员手工改数据文件。

## 整体结构：四个前端，一套后端

项目在架构上做了明确的分层：前端负责交互和展示，后端集中处理业务规则、数据存储和算法计算。沟通管道只有一条——REST API，所有端都走同一套接口，没有例外。

### 四个子应用，各司其职

前端用 React + TypeScript 构建了四个独立应用，共享一套 API 封装和类型定义（在 `web/shared/` 下），各自跑在不同的 Vite 开发端口上：

| 应用 | 目录 | 核心职责 |
|------|------|---------|
| 门户端 | `web/portal` | 统一登录页，按角色分流到各子应用 |
| 用户端 | `web/sanyuan-user` | 问卷采集、风险评估、干预方案、就医推荐、健康档案 |
| 医生端 | `web/doctor` | 患者群体统计、三病分析图表、个体档案与随访 |
| 管理端 | `web/administrator` | 医生/用户账号管理、健康文章 CRUD、医院信息维护 |

四个应用共享 `web/shared/` 下的工具包，统一的 HTTP 请求封装（自动注入 JWT）、鉴权逻辑、环境变量读取和异步数据 Hook。这样做的好处很直接：新增一个端或者调整某个接口时，不需要在各处重复修改。

UI 层面统一使用 Tailwind CSS + shadcn/ui（Radix UI 的 React 封装），保证了视觉一致性，也让组件可以在端之间直接复用。比如 `DiseaseRiskPropagationModule`（三病传播分析模块）同时出现在用户端和医生端，代码只有一份。

### 后端：FastAPI 单体，够用就好

后端选择 FastAPI 而不是拆微服务，是基于一个务实的判断：这个阶段的核心诉求是把功能完整跑通，而不是提前为"可能的高并发"买单。FastAPI 的异步支持足够应对当前规模，SQLAlchemy 对接 MySQL 让数据模型定义清晰可查。

后端只有一个入口文件 `web/backend/app/main.py`，约 3400 行，包含 50+ 个路由。严格来说这个文件偏大了，但按业务前缀分成了清晰的区块：`/api/portal/*`、`/api/user/*`、`/api/doctor/*`、`/api/admin/*`、`/api/risk/*`。后续可以按模块拆分成独立的 router 文件，不影响当前功能。

核心模块的职责划分：

| 模块 | 文件 | 职责 |
|------|------|------|
| 数据模型 | `models.py` | 7 张 SQLAlchemy 表：管理员、医生、用户档案、健康快照、文章、配图、医院 |
| 鉴权 | `portal_auth.py` + `auth_accounts.py` + `deps.py` | JWT 签发/验证、bcrypt 密码哈希、三角色依赖注入 |
| 问卷落库 | `questionnaire_save.py` | 字段映射、类型转换、衍生指标自动计算 |
| 风险引擎 | `risk_engine.py` | 三病概率计算、风险打分、传播分析、核心因子提取 |
| 内容推荐 | `guide_recommend.py` | 按疾病和风险等级匹配健康指南，关键词提权排序 |

## 数据怎么流动：从填问卷到看结果

整个系统的数据主线是一条清晰的链条：**问卷 → 落库 → 评估 → 推荐 → 展示**。每一步都是上一步的自然延续，没有旁路。

### 第一步：用户填问卷

用户端的数据采集页（`DataCollection.tsx`，约 97KB）把问卷拆成四个区块：基础信息（20 项）、生活习惯（11 项）、生理指标（24 项）和影像上传。用户也可以上传化验单照片，后端做 OCR 预填以减少手工输入。

这份问卷不是拍脑袋设计的，每个字段都对应后续风险评估模型需要的输入特征。比如"腰围"直接进入糖尿病 RF 模型的特征向量，"ALT/AST 比值"是肝病评估的关键衍生指标。

### 第二步：后端落库

前端提交后，`questionnaire_save.py` 接手。它做的事情不复杂但琐碎：把前端传来的嵌套 JSON 扁平化，做类型转换（字符串转数值、是/否转布尔），计算衍生指标（BMI、MAP、TyG、ALT/AST、TC/HDL 等），然后写入 `user_info` 表的 70+ 个字段中。

这里有一个设计选择值得说明：所有健康档案数据集中在 `user_info` 一张表里，而不是拆成"基础信息""生活习惯""体检指标"三张表。这样做的好处是查询简单，一次 `SELECT` 拿到全部数据，风险引擎不需要做 JOIN。代价是表很宽，但 70 列对 MySQL 来说完全在舒适区内。

### 第三步：风险评估

用户保存问卷后，前端自动触发 `POST /api/risk/predict`。`risk_engine.py` 读取用户的全量健康档案，分别计算三病风险：

- **糖尿病**：用随机森林 pipeline（`joblib` 持久化的 scikit-learn 模型）输出概率，同时基于全局特征重要性 CSV 计算各因子的贡献度，取 Top5 展示。
- **肝病**：采用启发式规则计算概率，因为当前训练数据量不足以支撑一个可靠的 ML 模型，规则反而比一个欠拟合的模型更可信。
- **脑卒中**：调用多模态融合预测模块（结构化数据 + 影像特征），在模型不可用时回退到启发式规则。这种"模型优先、规则兜底"的策略保证了演示环境即使没有 GPU 也能完整跑通。

三病评估结果汇总为"综合指数"（三病 score 的均值，0-100，越高越危险），在前端反向换算成"健康综合分"（100 - 综合指数）展示给用户，这个方向上的翻转是为了让用户直觉上"分数越高越好"。

除了概率和分数，风险引擎还输出"传播展示分"——模拟三病之间的相互影响关系，不是严格的因果推断，而是用几何耦合和因子余弦相似度做的一个演示性指标。它的价值在于让用户直观感受到"脂肪肝不控制可能推高糖尿病风险"，而不是只看到三个孤立的数字。

### 第四步：推荐干预方案

拿到风险等级之后，`guide_recommend.py` 按规则做内容匹配：

- 如果三病都是低风险，推所有认知类和饮食运动类文章。
- 如果某病是中或高风险，按病种和风险等级筛选文章，标题中包含病种关键词的优先展示。

这套规则足够简单，维护成本低，也预留了接入 AI 推荐的接口（`aiIntervention.ts` 已经在前端封装好了调用入口）。当前阶段用规则跑通流程比直接上模型更明智，先让内容团队把文章库建起来，再看推荐质量是否需要算法增强。

### 第五步：就医推荐

`GET /api/user/intervention/hospitals` 返回医院列表（当前是湛江三甲医院的静态数据，含经纬度）。前端用浏览器 Geolocation API 获取用户位置，Haversine 公式计算直线距离，按距离升序排列。

这个功能的定位是"演示路径"而非"生产级导航"——真实就医决策涉及科室匹配、医保覆盖、挂号可用性等复杂因素，当前阶段用直线距离排序足够让评审老师理解设计意图。医院数据从静态 JSON 起步，管理端已经提供了 CRUD 界面，过渡到动态维护只是把数据源从文件切到数据库的事。

## 几个关键取舍

做原型项目，最难的不是"做什么"，而是"不做什么"。下面是在开发过程中主动做出的选择，以及背后的原因。

### 风险计算集中放在后端

前端只负责传数据和画图表，不参与任何概率计算。这个决定省掉了多端重复实现的风险，用户端、医生端、管理端各自都有展示风险数据的界面，如果每端自己算一遍，用不了多久就会出现"同一份数据三个端显示三种结果"的口径漂移。后端统一计算、前端只做展示，把问题收敛到一个地方。

### 图片用磁盘存储而不是数据库 BLOB

健康文章配图和用户上传的医学影像（CT、超声、眼底照片）统一存在磁盘上，数据库只存文件路径。BLOB 方案的诱惑是把所有数据关在一个系统里方便备份，但医学影像动辄几十 MB，塞进数据库会让表体积快速膨胀、备份变慢、迁移麻烦。磁盘存储配合 `FileResponse` 返回，性能和可维护性都好得多。

### 统一门户做登录分流而不是各端各自登录

四个前端应用共享同一个登录入口（门户端），登录后 JWT 中携带角色信息，后端根据角色做权限校验。这样做有两个好处：一是用户体验统一，不需要记住"用户端登录页"和"医生端登录页"是两个不同的 URL；二是鉴权逻辑集中在后端，避免各端分别实现导致的安全漏洞。

### 单体后端而不是微服务

对于一个演示原型来说，微服务架构的收益远远抵不上它带来的复杂度。当前后端一台机器就能跑，所有业务逻辑在一个进程内调用，没有 RPC 延迟、没有分布式事务、不需要服务发现。如果将来某个模块（比如风险引擎）需要独立扩展，从单体中拆出它是明确的工程任务，不需要提前做。

### 规则推荐优先，AI 推荐做可插拔增强

健康指南推荐目前完全基于规则，按疾病标签和风险等级做筛选和排序。AI 推荐接口已经定义好（`/api/user/intervention/guides/ai-recommended`），前端也封装了调用入口（`aiIntervention.ts`），但后端当前返回的是"功能预留"状态。这个节奏是对的：内容库还在建设初期，AI 推荐没有足够数据做个性化，硬上反而显得演示不稳定。等文章量上来、用户行为数据积累到一定规模，再切入模型推荐就是水到渠成的事。

## 后续可以往哪些方向演进

当前版本把"能用"这件事做到了，但离"好用"还有距离。下面列出的是已经预留了接口、但不需要在演示阶段全部实现的演进方向。

### 风险引擎：从规则和单模型到多模态融合

目前糖尿病用 RF、肝病用规则、卒中做多模态融合，三病各自用不同的技术路线，不是因为设计上故意异构，而是各自的数据条件不同。后续三个方向都值得深入：

- 糖尿病可以接入更细粒度的时序模型（比如用连续多次体检数据做轨迹预测，而不是只看一次快照）。
- 肝病积累足够标注数据后可以用超声图像做端到端分类，替代当前的启发式规则。
- 卒中模型已经搭好了多模态框架，后续可以接入更高质量的结构化数据和 DICOM 影像。

### 内容推荐：从规则筛选到个性化排序

当前的推荐逻辑是"匹配疾病和风险等级"，本质上是布尔过滤。后续可以在过滤之后加一层排序，比如根据用户的阅读历史、年龄性别、生活方式偏好来做加权。技术上前端 `aiIntervention.ts` 和后端推荐接口都已经留好了扩展点。

### 内容与数据维护：从静态文件到管理端自治

医院数据目前是 `hospitals.json` 静态文件，健康文章通过 Excel 脚本导入。管理端已经提供了这两类数据的 CRUD 界面，后续只需要让后端优先读数据库而不是 JSON 文件，内容团队就能完全自助维护，不用再走"提需求 → 开发改文件 → 部署"的流程。

### 多端共享的进一步沉淀

当前四个前端已经共享了 `web/shared/` 下的 API 封装、鉴权逻辑和工具函数。后续可以进一步沉淀的内容包括：风险评估展示组件（用户端和医生端都在用）、等级阈值计算函数（目前首页和风险评估页各写了一份）、以及 Tailwind 主题配置。把共享层做厚，新增第五个端（比如"家属端"或"体检中心端"）的成本就会显著降低。

## 总结

HealthSystem 的设计思路可以概括为一句话：**用简单可靠的架构，把"早筛 → 干预 → 就医"这条主线完整跑通**。

它不是一个追求技术花哨的项目，单体后端、规则推荐、静态数据起步，每一步都选了当前阶段最务实的方案。但它也不是一个"做完就扔"的原型：数据模型、API 设计、模块拆分和扩展点预留，都为后续从原型走向产品留出了明确的路。

前端的四个子应用和后端的一套统一服务之间，通过清晰的 API 契约解耦；数据从问卷到评估再到推荐的流转路径，每一步都有明确的输入输出和兜底策略；关键取舍上优先保证口径统一和演示稳定性，而不是追逐"看起来更先进"的技术方案，这些选择放在一起，构成了这个项目最底层的设计逻辑。
